# SplitSmart — AI Usage Documentation

This document describes the AI tools, prompts, and debugging cycles used to co-author, debug, and deploy the SplitSmart Shared Expenses Manager.

---

## 🛠️ 1. AI Tools & Agents Used

During this project, we operated as a multi-agent system powered by the **Antigravity** framework:
1.  **Antigravity Core AI**: Orchestrated the architectural planning, resolved dependency issues, designed database structures, managed environment variables, and configured the live Vercel/Neon deployment.
2.  **`research` Subagent**: Used to analyze Next.js 15/16 configurations, NextAuth v5 middleware compatibility patterns, and Prisma 7 database adapter setups.
3.  **`core_logic_builder` Subagent**: Responsible for co-authoring the business logic modules (`balance-engine.ts`, `anomaly-detector.ts`, and `currency.ts`).
4.  **`ui_builder` Subagent**: Assisted in scaffolding component outline modules and dashboard layouts.

## 🐞 3. Concrete Cases Where the AI Generated Errors & Resolutions

### Case 1: NextAuth v5 Middleware Edge Runtime Crash
*   **What the AI Did Wrong**: The code generated by the AI imported the full NextAuth instance from `src/lib/auth.ts` directly into `src/middleware.ts`. Since `auth.ts` references the Prisma database client adapter (which relies on native Node.js socket packages like `net`, `dns`, and `tls`), these native modules were bundled into the Next.js Middleware. Next.js Middleware runs in the Edge runtime, which throws compilation/runtime exceptions when native Node modules are imported.
*   **How We Caught It**: The compilation failed during `npm run build` and threw exceptions:
    ```text
    Module not found: Can't resolve 'net' in 'node_modules/pg'
    ```
*   **What We Changed**: We separated the configuration. We created `src/lib/auth.config.ts` containing strictly Edge-compatible auth parameters (like the Credentials provider login callbacks). We then updated `src/middleware.ts` to only import this lightweight configuration, while reserving database-dependent auth initializations exclusively for `src/lib/auth.ts` (which is never run in the Edge runtime).

---

### Case 2: Email Case Sensitivity during Login Queries
*   **What the AI Did Wrong**: The generated Credentials provider verification callback in `src/lib/auth.ts` executed a direct case-sensitive email lookup:
    ```typescript
    const user = await prisma.user.findUnique({ where: { email } });
    ```
*   **How We Caught It**: During user testing, trying to log in with capitalized names (e.g. `Aisha@splitsmart.app`) failed and redirected back to `/login` with invalid credentials errors, despite the account existing in the database (stored as lowercase `aisha@splitsmart.app` by the seed script).
*   **What We Changed**: We updated the `authorize` callback in `src/lib/auth.ts` to explicitly normalize the input email using `.toLowerCase().trim()` before executing the database query:
    ```typescript
    const normalizedEmail = credentials.email.toLowerCase().trim();
    const user = await prisma.user.findUnique({ where: { email: normalizedEmail } });
    ```

### Case 3: Recharts SSR Hydration / Rendering Mismatch
*   **What the AI Did Wrong**: The AI generated dashboard charts (AreaChart, PieChart) using Recharts inside standard Next.js Server Components without declaring the `"use client"` directive.
*   **How We Caught It**: Next.js threw console hydration errors and build failures indicating that Recharts attempts to read browser APIs (such as `window.innerWidth` or SVG container measurements) which do not exist during server-side pre-rendering.
*   **What We Changed**: We added `"use client"` at the top of the charting files:
    *   `spending-area-chart.tsx`
    *   `category-donut-chart.tsx`
    *   `top-spenders-bar-chart.tsx`
    We also wrapped the Recharts components in responsive width wrappers and initialized them inside a client-side `useEffect` mount check to prevent HTML hydration mismatches.

---

### Case 4: Prisma 7 Datasource Schema Definition
*   **What the AI Did Wrong**: Generated the `schema.prisma` file with the connection string `url` hardcoded or mapped via `env("DATABASE_URL")` under the `datasource` block, using patterns deprecated or altered by Prisma 7's adapter model.
*   **How We Caught It**: Prisma CLI commands (`npx prisma validate` / `npx prisma db push`) failed with syntax and source schema configuration errors.
*   **What We Changed**: Adjusted `prisma/schema.prisma` to conform to Prisma 7 configuration specifications. We removed direct runtime database driver declarations from the schema and created a dedicated `prisma.config.ts` file to wire up the adapter pool configurations:
    ```typescript
    // prisma.config.ts
    export default {
      datasource: {
        url: process.env.DATABASE_URL
      }
    }
    ```
